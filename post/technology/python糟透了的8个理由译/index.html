偶尔我和一些技术朋友一起出去吃午饭, 我们一起度过了愉快的时光. 我们谈论项目, 时事和各种技术相关问题. 不可避免地, 讨论将转向编程语言. 有人可能会感叹"我必须修改一些Java代码. 我讨厌Java.(哦, 对不起, Kyle.)” (十年前我们给Kyle起了绰号"Java-boy"可能没有帮助.)另一个将抱怨一些没有人想要重写的旧的单片shell代码.

而我, 好吧… 我只是脱口而出: 我讨厌Python. 我热情地讨厌它. 如果我可以选择使用一些预先存在的Python代码或者用C语言重写它, 我宁愿用C语言重写它.

当我完成喊叫时, 比尔幽默地补充道, “但是你真的怎么做? 想想Python, Neal?” 所以我将这篇博客文章献给了Bill.

这是我列出的"Python糟透了的8个原因”.

原因1: 版本

如果您安装了默认的Linux操作系统, 那么它很可能会安装多个版本的Python. 它可能有Python2和Python3, 甚至可能还有一些分数版本, 如3.5或3.7. 这是有原因的: Python3与Python2不完全兼容. 甚至一些分数版本也足够明显, 缺乏向后兼容性.

我全都是为语言添加新功能. 我甚至不介意一些旧版本是否过时. 但是, Python安装在单独的安装中. 我的Python 3.5代码不适用于Python 3.7安装, 除非我故意将其移植到3.7. 足够的Linux开发人员已经决定移植是不值得的, 因此Ubuntu安装了Python2和Python3 - 因为它们是不同核心功能所需要的.

缺乏向后兼容性和拆分版本通常是丧钟. Commodore创建了第一台家用电脑(早在IBM PC或Apple之前). 但Commodore PET与随后的Commodore CBM计算机不兼容. 并且CBM与VIC-20, Commodore-64, Amiga等不兼容. 因此, 您要么花费大量时间将代码从一个平台移植到另一个平台, 要么您放弃了平台.(今天的Commodore在哪里? 当用户放弃平台时, 它就消失了.)

同样, Perl曾经很受欢迎. 但是当Perl3问世时, 它并没有完全向后兼容很多Perl2代码. 社区抱怨, 好的代码被移植, 其余的被放弃了. 然后是Perl4, 同样的事情发生了. 当Perl5问世时, 许多人只是转而使用更稳定的不同编程语言. 今天, 只有一小部分人积极使用Perl来维护现有的Perl项目. 我没有看到任何基于Perl的重大新项目.

通过相同的方式, Python为每个版本都有不同的代码库. 社区一直拖着旧版本. 所以你最终得到了很多旧的, 死的Python代码, 因为没有人愿意花时间把它移植到最新版本. 据我所知, 没有人为Python2创建新代码, 但是我们拖延它, 因为没有人将所需的代码移植到Python3.x. 在官方Python网站上, 他们的文档被主动维护并可用于Python 2.7, 3.5, 3.6和3.7 - 因为他们无法决定放弃旧代码. Python就像编程语言的僵尸 - 死者只是继续前行.

原因2: 安装

使用大多数软件包, 您可以轻松运行apt, yum, rpm或其他一些安装基础并获取最新代码. Python的情况并非如此. 如果使用’apt-get install python’进行安装, 则不知道实际安装的是哪个版本, 并且可能与您需要的所有代码不兼容.

因此, 您需要安装所需的Python版本. 对于我所参与的其中一个项目, 我们使用了Python. 但是我们不得不使用Python3.5(当时最新的). 我的计算机最终安装了Python2, Python2.6, Python3和Python3.5. 其中两个来自操作系统, 一个来自项目, 一个来自我因某些其他原因安装了一些不相关的软件. 尽管它们都是"Python”, 但它们并非完全相同.

如果你想安装Python的软件包, 你应该使用"pip”.(Pip代表"Pip Installs Packages”, 因为有人认为递归缩略词仍然很有趣.)但是由于系统上有许多版本的Python, 你必须记住使用正确版本的pip. 否则, ‘pip’可能会运行’pip2’而不是你需要的’pip3.7’.(如果名称不存在, 你需要指定pip3.7的实际路径.)

我被一个队友告知我需要配置我的环境, 以便所有东西都使用Python 3.5基础. 这很有效, 直到我开始第二个需要Python 3.6的项目. 有两个不同版本的Python的两个并发项目 - 不, 这并不令人困惑.(什么是讽刺的表情?)

pip安装程序将文件放在用户的本地目录中. 您不使用pip来安装系统范围的库. 并且Gawd禁止你犯下运行’sudo pip’的错误, 因为这会搞砸整个电脑! 运行sudo可能会使某些软件包在系统级别安装, 某些软件包安装用于错误版本的Python, 并且主目录中的某些文件可能最终归root用户所有, 因此未来的非sudo pip安装可能会因权限而失败. 只是不要这样做.

顺便说一句, 谁维护这些pip模块? 社区. 也就是说, 没有明确的所有者, 也没有强制性的起源或责任链. 今年早些时候, 发现一个版本的PyPI有一个后门偷了SSH凭据. 这根本不会让我感到惊讶.(出于同样的原因, 我不使用Node.js和npm; 我不相信他们的社区存储库.)

原因3: 语法

我是可读代码的坚定信徒. 乍一看, Python似乎非常易读. 也就是说, 直到你开始制作大型代码库.

大多数编程语言使用某种表示法来标识范围 - 函数的开始和结束, 条件语句中包含的操作, 变量定义的范围等. 使用C, Java, JavaScript, Perl和PHP, 大括号 {...} 定义范围. Lisp使用括号 (...) . 那么Python? 它使用空格. 如果需要为复杂代码定义范围, 则缩进接下来的几行. 当缩进结束时, 范围结束.

Python手册说你可以使用任意数量的空格或制表符用于定义范围. 但是, 总是使用四个空格! 如果要缩进两次以进行嵌套, 请使用八个空格! Python社区已经对这个术语进行了标准化, 即使它不在Python手册中. 忘记文档中的示例使用 tabs , tabs +1空格 和其他缩进的事实. 社区对使用四个空间感到愤怒. 因此, 除非您打算永远不向其他任何人展示您的代码, 否则每个缩进都要使用四个空格.

当我第一次看到Python代码时, 我认为使用缩进来定义范围似乎是一个好主意. 然而, 这是一个巨大的缺点. 允许深度嵌套, 但是线条可以变宽以至于它们在文本编辑器中包裹线条. 长函数和长条件操作可能使得难以匹配开始到结束. 而且我可怜任何错误空格并且意外地放入三个空格而不是四个空格的人 - 这可能需要数小时来调试和追踪.

对于其他语言, 我已经养成了调试代码而没有任何缩进的习惯. 通过这种方式, 我可以快速浏览代码, 并在完成后轻松识别和删除调试代码. 但是用Python? 任何没有缩进的东西都会产生缩进错误. 这意味着调试代码必须融入活动代码.

原因4: 包括

大多数编程语言都有一些方法可以包含其他代码块. 对于C, 它是”#include”. 对于PHP, 有’include’, ‘include_once’, ‘require’和’require_once’. 对于Python, 有"import”.

Python的导入允许包括整个模块, 模块的一部分或模块中的特定功能. 查找可导入的列表是非直观的. 使用C, 您可以查看/usr/include/*.h. 但是用Python? 最好使用’python -v’列出它看起来的所有位置, 然后搜索该列表中每个目录和子目录中的每个文件. 我有喜欢Python的朋友, 我通过标准模块看到他们grep, 因为他们正在寻找他们想要导入的东西. 认真地.

导入功能还允许用户重命名导入的代码. 它们基本上定义了一个自定义命名空间, 乍一看, 这似乎很不错, 但最终会影响可读性和长期支持. 重命名模块非常适合小脚本, 但对于长程序来说真的很糟糕. 使用1-2个字母名称空间的人, 例如"import numpy as n"应该被枪毙(或强制将所有代码转换为Perl5).

但这不是最糟糕的部分. 对于大多数语言, 包括代码只包括代码. 如果有一个带有构造函数的全局对象, 一些语言(如面向对象的C++)可以执行代码. 类似地, 一些PHP代码可能会定义全局变量, 因此导入可以运行代码 - 但这通常被认为是一种不好的做法. 相比之下, 许多Python模块包括在导入期间运行的初始化函数. 你不知道什么在运行, 你不知道它在做什么, 你可能没注意到. 除非存在命名空间冲突, 否则在这种情况下, 您可以花很多时间来追踪原因.

原因5: 命名法

在其他所有语言中, 数组都称为"数组”. 在Python中, 它们被称为"列表”. 关联数组有时称为’hash’(Perl), 但Python称之为’字典’. Python似乎没有使用在整个计算机和信息科学领域发现的常用术语.

然后是库的名称. PyPy, PyPi, NumPy, SciPy, SymPy, PyGtk, Pyglet, PyGame … (是的, 前两个发音方式相同, 但是它们做的事情非常不同.)我知道’py’适用于Python. 但他们不能确定它是第一还是第二?

一些常见的库只是放弃了类似双关语的"Py"命名约定. 这包括matplotlib, nose, Pillow和SQLAlchemy. 虽然一些名称可能会给你一个暗示目的(例如, “SQLAlchemy"包含SQL, 所以它可能是一个SQL接口), 其他名称只是随机单词. 如果您不知道"BeautifulSoup"的作用, 您能从名称中看出它是一个HTML/XML解析器吗?(顺便说一句, BeautifulSoup有很好的文档和易于使用. 如果每个Python模块都是这样的, 我不会抱怨太多. 不幸的是, 这是例外而不是常态. 大多数Python库严重抄袭文档. )

总的来说, 我将Python视为具有可怕且不一致的命名约定的库的集合. 我有一个[]常设的抱怨](https://www.hackerfactor.com/blog/index.php?/archives/415-Open-Source-Sucks.html), 开源项目通常有可怕的名字. 除非你知道这个项目, 否则你永远不知道它的名字是什么. 除非你知道要寻找什么, 否则你很可能只会偶然发现它在路过的时候绊倒了. Python的大多数库强化了这种负面批评.

原因6: 怪癖

每种语言都有它的怪癖. 使用C, 使用＆和*来访问地址空间和值是奇怪的命名法. C也有使用 ++ 和 -- 的递增/递减快捷方式. 使用Bash, 当引用括号和正则表达式的句点等特殊字符时, 有个"when to use a backslash”. JavaScript存在兼容性问题(并非每个浏览器都支持所有有用的功能). 然而, Python比我见过的任何其他语言都有更多怪癖. 考虑字符串:

  • 在C中, 双引号括起字符串. 单引号括起来的字符.
  • 在PHP和Bash中, 两种类型的引号都可以包含字符串. 但是, 双引号可以在字符串中嵌入变量. 相比之下, 单引号字符串是文字; 任何类似嵌入式变量的名称都不会扩展.
  • 在JavaScript中, 单引号和双引号之间确实没有区别.
  • 在Python中, 单引号和双引号之间没有区别. 但是, 如果您希望字符串跨越行, 则需要使用三引号 """string""" 或 '''string''' . 如果你想使用二进制文件, 那么你需要用 b（b'binary'） 或 r（r'raw'） 来优先选择字符串. 有时您需要使用 str（string） 将字符串转换为字符串, 或使用 string.encode（'utf-8'） 将其转换为utf8.

如果你认为 = , == 和 === 最初在PHP和JavaScript中有点奇怪, 那么就等你在Python中使用引号玩耍吧.

原因7: 按对象引用传递

大多数编程语言都按值传递函数参数. 如果函数改变了值, 则结果不会传递回调用代码. 但正如我已经解释过的那样, Python不同寻常. Python默认使用 pass-by-object-reference 参数执行函数. 这意味着更改源变量可能最终会更改值.

这是程序, 功能和面向对象编程语言之间的巨大差异之一. 如果每个变量都是通过对象引用传递的, 并且对变量的任何更改都会在任何地方更改引用, 那么您也可以使用全局变量来处理所有内容. 使用不同的名称调用同一个对象不会更改对象, 因此它实际上是全局的. 正如C程序员很久以前所学到的, 全局变量是邪恶的, 不应该使用.

在Python中, 您必须按值传递变量. 说 a = b 只是为同一个对象空间指定另一个名称; 这不会将b的值复制到a中. 如果您确实要复制该值, 则需要使用复制功能. 通常这是 a = b.copy() . 但是, 请注意我说"通常”. 并非所有数据类型都具有"复制"原型. 或者复制功能可能不完整. 在这些情况下, 您可以使用一个名为 copy 的独立库: a = copy.deepcopy(b) .

原因8: 本地名称

在使用库或函数之后命名程序是一种常见的编程技术. 例如, 如果我正在使用名为 libscreencapture.so 的C库测试屏幕捕获程序, 我会调用我的程序 screencapture.c 并编译成 screencapture.exe .

gcc -o screencapture.exe screencapture.c -lscreencapture

使用C, Java, JavaScript, Perl, PHP等, 这很好用, 因为语言可以很容易地将资源库与本地程序区分开来. 他们有不同的路径. 但是用Python? 不要这样做. 永远不要这样做. 为什么? Python假设您首先要导入本地代码. 如果我有一个名为 screencapture.py 的程序使用 import screencapture , 那么它将导入自己而不是系统库. 至少, 您应该调用本地程序 myscreencapture.py

并非一切都不好

Python是一种非常流行的语言, 拥有庞大的追随者. 我甚至有一些非常喜欢Python的朋友 - 这是他们首选的编程语言. 多年来, 我和他们一起讨论过这些问题, 每次他们点头表示同意. 他们并不反对这些是Python的问题; 他们只是认为他们不再喜欢这种语言并不够糟糕.

我的朋友经常引用所有存在的非常酷的Python库. 我同意一些库非常有用. 例如, BeautifulSoup是我用过的最好的HTML解析器之一, NumPy使多维数组和复杂的数学更容易实现, 而TensorFlow对于机器学习非常有用. 但是, 我不会因为喜欢TensorFlow或SciPy而在Python中创建一个单片程序. 我不会放弃free pony的可读性和可维护性; 这不值得努力.

通常当我写一个关于某个主题的负面批评时, 我也会尝试写一些积极的东西. 我跟着我的博客文章Open Source Sucks和Open Source Rocks.FFmpeg, 我明确提到它是如何最好的视频处理库. 但我无法列出关于Python的好东西, 因为我真的认为Python很糟糕.
